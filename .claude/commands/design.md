# /design - 기능 설계 스킬

사용자의 기능 요청을 받아 설계하고, Sonnet이 바로 구현할 수 있는 수준의 상세 구현 계획서를 작성한다.

## 입력

$ARGUMENTS

## 실행 흐름

### 1. 요청 분석

사용자 요청을 파악한다. 추가 정보가 필요하면 2~3개씩 질문한다.
질문이 불필요할 만큼 명확하면 바로 다음 단계로 넘어간다.

### 2. 규모 판단

요청의 규모를 판단한다:

- **소규모** (파일 1~2개, 로직 변경 최소): 이슈/브랜치/PR 없이 직접 처리 후 커밋
- **중규모 이상** (파일 3개+, 새 기능, 구조 변경): 아래 전체 프로세스 실행

소규모로 판단하면 사용자에게 알리고 바로 구현한다. 커밋 메시지는 `docs/conventions.md`의 커밋 컨벤션을 따른다.

### 3. GitHub 이슈 생성 (중규모 이상)

```bash
gh issue create --title "<커밋 컨벤션 형식 제목>" --body "<설명>" --label "<라벨들>"
```

라벨은 `docs/conventions.md`의 GitHub 라벨 가이드를 참조한다.

### 4. 브랜치 생성

```bash
git checkout -b <type>/<이슈번호>-<간단한설명>
```

브랜치 네이밍은 `docs/conventions.md`의 브랜치 전략을 따른다.

### 5. 코드베이스 탐색

구현에 필요한 파일들을 읽고 분석한다:
- 관련 기존 코드 파악
- 재사용 가능한 함수/컴포넌트 확인
- 영향 받는 파일 목록 정리
- `docs/conventions.md`의 규칙 확인

### 6. 상세 구현 계획서 작성

`.claude/plans/` 디렉토리에 계획서를 저장한다:

```
.claude/plans/<이슈번호>-<간단한설명>.md
```

계획서에 포함할 내용:

```markdown
# [이슈 제목]

## 이슈
- 번호: #N
- 브랜치: <브랜치명>

## 개요
[변경 목적과 배경 1~2줄]

## 변경 파일 목록
| 파일 | 변경 유형 | 설명 |
|------|----------|------|
| `src/lib/naver/auth.ts` | 신규 생성 | OAuth 토큰 생성 |
| `src/app/api/orders/route.ts` | 신규 생성 | 주문 조회 API |

## 구현 상세

### 1. [작업 단위 1]

**Before:** (기존 코드가 있는 경우)
```typescript
// 현재 코드
```

**After:**
```typescript
// 변경될 코드
```

**설명:** [왜 이렇게 변경하는지]

### 2. [작업 단위 2]
...

## 커밋 계획
1. `feat(scope): 첫 번째 커밋 내용` - [포함 파일들]
2. `feat(scope): 두 번째 커밋 내용` - [포함 파일들]

## 테스트 계획
- [ ] [테스트 케이스 1]
- [ ] [테스트 케이스 2]

## 체크리스트
- [ ] `docs/conventions.md` 규칙 준수
- [ ] `.env.local`에 민감 정보 하드코딩 없음
- [ ] 타입 안전성 확인 (any 없음)
- [ ] 에러 핸들링 포함
```

### 7. 사용자 확인

계획서 요약을 보여주고 확인을 받는다. 수정 요청 시 반영한다.

### 8. 완료 안내

```
설계 완료!

계획서: .claude/plans/<파일명>.md
이슈: #N
브랜치: <브랜치명>

구현을 시작하려면:
/compact → /model sonnet → /build
```

## 주의사항

- 설계 단계에서는 코드를 직접 작성하지 않는다 (소규모 제외)
- 계획서는 Sonnet이 추가 질문 없이 구현할 수 있을 만큼 상세하게
- Before/After 코드 예시는 실제 프로젝트 코드 기반으로 작성
- 기존 코드와의 일관성을 최우선으로 고려
- `docs/project-history.md`에 기록할 내용도 계획서에 포함
