# /build - 기능 구현 스킬

`.claude/plans/`에 저장된 설계 계획서를 로드하고, 계획대로 정확히 구현한다.

## 입력

$ARGUMENTS

인자가 없으면 `.claude/plans/` 디렉토리에서 가장 최근 계획서를 자동으로 선택한다.

## 실행 흐름

### 1. 계획서 로드

```bash
ls -lt .claude/plans/*.md
```

인자가 있으면 해당 파일을, 없으면 최신 파일을 읽는다.
계획서가 없으면 "설계 계획서가 없습니다. /design을 먼저 실행해주세요." 출력 후 종료.

### 2. 현재 상태 확인

```bash
git status
git branch --show-current
```

- 계획서에 명시된 브랜치에 있는지 확인
- 다른 브랜치에 있으면 올바른 브랜치로 전환
- 커밋되지 않은 변경이 있으면 사용자에게 알림

### 3. 구현

계획서의 "구현 상세" 섹션을 순서대로 실행한다:

- 각 작업 단위별로 코드를 작성
- **계획서에 없는 변경은 하지 않는다** (scope creep 방지)
- 계획서와 다른 방식이 더 나은 경우, 사용자에게 먼저 물어본다
- `docs/conventions.md`의 규칙을 준수한다

### 4. 커밋

계획서의 "커밋 계획"에 따라 커밋한다:

- 커밋 메시지는 `docs/conventions.md`의 Conventional Commits 형식
- 각 커밋은 논리적으로 하나의 단위

### 5. 코드 리뷰 (자동 실행)

PR 생성 전에 이 브랜치의 전체 변경사항을 리뷰한다. `/review-code` 스킬과 동일한 수준으로 실행.

```bash
git diff main...HEAD
```

#### 보안 감사 (최우선 — CRITICAL 발견 시 즉시 수정 후 커밋)

| 점검 항목 | 심각도 |
|----------|--------|
| 크리덴셜 하드코딩 (API 키, 비밀번호, 토큰) | CRITICAL |
| `.env.local` 파일이 커밋에 포함됨 | CRITICAL |
| SQL Injection 가능성 (raw query 사용) | HIGH |
| 외부 입력 미검증 | HIGH |
| 민감 정보 로그 출력 | MEDIUM |
| 에러 메시지에 내부 정보 노출 | LOW |

#### 타입 안전성
- `any` 사용 여부
- 외부 데이터(API 응답) zod 파싱 여부
- 타입 단언(`as`) 최소화 여부

#### 구조/설계
- 레이어 분리 준수 (API 라우트 → lib/ 서비스 → DB)
- 함수 단일 책임 원칙
- API 라우트에서 직접 DB 쿼리 없음

#### 컨벤션
- 네이밍 규칙 (PascalCase, camelCase, UPPER_SNAKE_CASE)
- import 정렬
- 매직 넘버 없음 (상수로 추출)

#### 리팩토링 판단
- 코드 중복 3회 이상 → 공통 함수 추출 ("지금 고칠 것" 표시)
- 함수 50줄 초과 → 분리 ("지금 고칠 것" 표시)
- 파일 300줄 초과 → 모듈 분리 ("나중에 고칠 것" 표시)

#### 리뷰 결과 출력 형식

```markdown
## 코드 리뷰 결과

### 보안 감사
- [PASS/FAIL] 크리덴셜 하드코딩: [결과]
- [PASS/FAIL] 입력 검증: [결과]
- ...

### 코드 품질
- [PASS/WARN] 타입 안전성: [설명]
- [PASS/WARN] 구조/설계: [설명]
- ...

### 리팩토링 제안
1. [지금] [파일:라인] [제안 내용]
2. [나중에] [파일:라인] [제안 내용]

### 요약
- 보안 이슈: N개 (CRITICAL: N, HIGH: N)
- 품질 경고: N개
- 리팩토링 제안: N개 (즉시: N, 나중: N)
```

리뷰 결과에 따라:
- CRITICAL/HIGH 보안 이슈 → 즉시 수정 후 커밋, 리뷰 재실행
- "지금 고칠 것" 리팩토링 → 수정 후 커밋
- "나중에 고칠 것" → PR 본문에 TODO로 기록

### 6. 테스트 실행

계획서의 "테스트 계획"에 따라:

```bash
npx vitest run  # 테스트가 있는 경우
```

테스트 실패 시 수정 후 커밋.

### 7. PR 생성

```bash
git push -u origin <브랜치명>
gh pr create --title "<커밋 컨벤션 제목>" --body "..."
```

PR 본문 형식:
```markdown
## 관련 이슈
Closes #N

## 변경 내용
- 변경사항 1
- 변경사항 2

## 코드 리뷰 결과
- [x] 보안 감사 통과 (CRITICAL: 0, HIGH: 0)
- [x] 타입 안전성 확인
- [x] 컨벤션 점검 완료
- [x] 코드 품질 확인

### 리팩토링 TODO (나중에 고칠 것)
- [ ] [파일:라인] [내용] (있는 경우에만 포함)

## 테스트
- [x] 테스트 케이스 1
- [x] 테스트 케이스 2
```

### 8. 프로젝트 히스토리 업데이트

`docs/project-history.md`에 해당 Phase/기능의 완료 기록을 추가한다.
계획서에 히스토리 기록 내용이 있으면 그대로 사용.

### 9. 완료 보고

```
구현 완료!

PR: #N (URL)
커밋: N개
코드 리뷰: 통과
테스트: 통과

다음 작업이 있으면 /design으로 새 기능을 설계하세요.
```

## 주의사항

- **계획서에 없는 코드를 작성하지 않는다.** 추가 변경이 필요하면 사용자에게 물어본다.
- 계획서의 Before/After 코드와 실제 코드가 다르면 (다른 PR에서 변경됨), 현재 코드 기준으로 판단하되 사용자에게 알린다.
- 코드 리뷰에서 보안 이슈 발견 시 반드시 수정 후 진행.
- 커밋 3~5개가 쌓이면 PR 생성을 고려한다.
