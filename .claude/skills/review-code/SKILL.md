# /review-code - 코드 리뷰 스킬

현재 브랜치의 변경사항에 대해 보안 감사 + 코드 품질 리뷰 + 컨벤션 점검 + 리팩토링 제안을 수행한다.

## 입력

$ARGUMENTS

인자가 없으면 현재 브랜치의 `main` 대비 전체 변경사항을 리뷰한다.
파일 경로가 주어지면 해당 파일만 리뷰한다.

## 실행 흐름

### 1. 변경 범위 파악

```bash
git diff main...HEAD --stat
git diff main...HEAD
```

변경된 파일 목록과 diff를 확인한다.

### 2. 보안 감사 (최우선)

가장 먼저 보안 이슈를 점검한다:

| 점검 항목 | 심각도 |
|----------|--------|
| 크리덴셜 하드코딩 (API 키, 비밀번호, 토큰) | CRITICAL |
| `.env.local` 파일이 커밋에 포함됨 | CRITICAL |
| SQL Injection 가능성 (raw query 사용) | HIGH |
| 외부 입력 미검증 | HIGH |
| 민감 정보 로그 출력 | MEDIUM |
| 에러 메시지에 내부 정보 노출 | LOW |

CRITICAL/HIGH 이슈 발견 시 즉시 사용자에게 알린다.

### 3. 코드 품질 리뷰

`docs/conventions.md`의 규칙을 기준으로 점검:

#### 타입 안전성
- `any` 사용 여부
- 외부 데이터 zod 파싱 여부
- 타입 단언 최소화 여부

#### 구조/설계
- 함수 단일 책임 원칙
- 레이어 분리 (API → lib/ → DB)
- 적절한 추상화 수준

#### 가독성
- 네이밍 규칙 준수
- 매직 넘버 여부
- 불필요한 주석 여부
- import 정렬

### 4. 리팩토링 제안

`docs/conventions.md`의 리팩토링 기준에 따라:

- 코드 중복 3회 이상 → 공통 함수 추출 제안
- 함수 50줄 초과 → 분리 제안
- 파일 300줄 초과 → 모듈 분리 제안
- API 라우트에서 직접 DB 쿼리 → lib/ 추출 제안

각 제안에 "지금 고칠 것" vs "나중에 고칠 것" 판단을 포함한다.

### 5. 리뷰 결과 출력

```markdown
## 코드 리뷰 결과

### 보안 감사
- [PASS/FAIL] 크리덴셜 하드코딩: [결과]
- [PASS/FAIL] 입력 검증: [결과]
- ...

### 코드 품질
- [PASS/WARN] 타입 안전성: [설명]
- [PASS/WARN] 구조/설계: [설명]
- ...

### 리팩토링 제안
1. [지금] [파일:라인] [제안 내용]
2. [나중에] [파일:라인] [제안 내용]

### 요약
- 보안 이슈: N개 (CRITICAL: N, HIGH: N)
- 품질 경고: N개
- 리팩토링 제안: N개 (즉시: N, 나중: N)
```

### 6. 수정 제안

CRITICAL/HIGH 보안 이슈가 있으면 수정 코드를 제안한다.
사용자가 수정을 원하면 바로 적용하고 커밋한다.

## 주의사항

- 보안 이슈는 반드시 먼저 확인하고 CRITICAL은 즉시 보고
- 리뷰는 `docs/conventions.md` 기준으로만 판단 (개인 취향 배제)
- 리팩토링 제안은 현재 규모에 맞는지 판단 포함
- 칭찬할 부분도 언급 (좋은 패턴, 깔끔한 구현)
